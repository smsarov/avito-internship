# Тестовое задание Frontend, весна 2026

Приложение запущено на VPS и доступно по [ссылке](http://80.93.63.231/ads)

## Стек технологий

- **React** 19, **TypeScript**
- **Vite**
- **react-router-dom**
- **TanStack Query**
- **Axios**
- **Zustand**
- **react-hook-form**, **Zod**
- **Tailwind CSS**
- **shadcn**
- **ESLint**, **Prettier**

Для проекта выбран `shadcn/ui`, потому что он построен на Radix UI (хороший a11i из коробки), легко кастомизируется, есть возможность добавлять компоненты  выборочно прямо в проект. `shadcn` компоненты находятся в [папке](/client/src/components/ui). Компонент Markdown (все, что в [blocks](/client/src/components/blocks/), [editor](/client/src/components/editor/) и [markdown](/client/src/components/markdown/)) также относится к `shadcn`. 

`Tailwind` выбран для ускорения написания стилей (и потому что компоненты `shadcn` работают именно с ним)

`Zustand`, потому что он не требует бойлерплейта, стейт менеджмент показался уместным  только на одной странице (/ads), а сложная логика с middleware и транзакциями не потребовалась.

### Инфраструктура

- **Docker**, **Docker Compose** — контейнеризация клиента и сервера

## Дополнительный функционал

- смена layout сетка/список на главной странице
- динамический title страниц

## Изменения в серверном коде

Я старался вносить минимальные изменения в сервер

1. `get \items` возвращает `id` объявлений
   - без этого невозможно было бы переходить на страницу промотра объявление по клику в списке

2. **CORS** — разрешены запросы с любого origin (`Access-Control-Allow-Origin:` * и сопутствующие заголовки)
   - по-хорошему надо сделать строже, но в изначальном варианте блокировался `put` запрос

3. **Два новых метода** для AI-функций:
  - `POST /items/suggestions/description` — генерация/улучшение описания;
  - `POST /items/suggestions/price` — предложение рыночной цены.

4. **Интеграция с GigaChat** — вызовы модели через пакет `gigachat` и переменную окружения `GIGACHAT_CREDENTIALS` (см. ниже). Без настроенных учётных данных эти маршруты отвечают `503`.

## Запуск из исходного кода

Предполагается установленный **Node.js 20+** и **npm**.

### 1. Получение репозитория

```bash
git clone https://github.com/smsarov/avito-internship.git
cd avito-internship
```

### 2. Запуск сервера

```bash
cd server
npm install
```

создайте файл `server/.env` и заполните его на основе примера:

```bash
cp .env.example .env
```

В `.env` укажите:

- `GIGACHAT_CREDENTIALS` — GigaChat Authorization key (строка в кавычках). Без него сервер запустится, но AI-маршруты вернут ошибку конфигурации. Подбробнее про то, как его получить можно найти [здесь](https://developers.sber.ru/docs/ru/gigachat/individuals-quickstart) и [здесь](https://developers.sber.ru/docs/ru/gigachat/quickstart/ind-create-project).
- `PORT` — по желанию (по умолчанию **8080**).

Запуск:

```bash
npm start
```

API будет доступен по адресу `http://localhost:8080` (или по выбранному `PORT`).

### 3. Запуск клиента

из корня проекта:

```bash
cd client
npm install
```

По умолчанию клиент обращается к API по `http://localhost:8080` (см. `VITE_API_URL` в [файле](/client/src/lib/axios.ts)). Если сервер на другом хосте или порту, задайте переменную перед запуском:

```bash
VITE_API_URL=http://127.0.0.1:8080 npm run dev
```

Запуск в режиме разработки:

```bash
npm run dev
```

Клиент будет доступен на `http://localhost:5173`

## Запуск через Docker

Нужны установленные **Docker** и **Docker Compose**.

Из **корня репозитория**:

```bash
docker compose up --build
```

- Клиент: **порт 80** (например, `http://localhost`).
- Сервер: **порт 8080** (`http://localhost:8080`).

Файл `server/.env` подхватывается Compose при наличии (см. `docker-compose.yml`). Для работы AI на бэкенде задайте в `server/.env` переменную `GIGACHAT_CREDENTIALS`.

Сборка фронтенда использует `VITE_API_URL` (по умолчанию `http://localhost:8080`). Если API вынесен на другой URL, передайте при сборке, например:

```bash
VITE_API_URL=https://api.example.com docker compose up --build
```

## Принятые решения

### Драфт

Изначально было не совсем понятно, как должны работать драфты при редактировании объявления. В итоге решил сделать так:

С сервера приходит объект объявления, в котором есть, помимо прочего, поле updated_at. Если драфта в localstorage нет, заполняем форму данными с сервера. Если он есть, то смотрим на updated_at. В случае, если драфт обновлен позже, чем данные сервера, считаем их устаревшими и показываем данные из запроса, иначе используем сам драфт. Он очищается автоматически при успешном submit и по нажатию на кнопку отмены.

### AI

Долго думал над тем, как организовать этот функционал. В частности, выделять ли его в отдельную feature или нет. В итоге решил оставить его как часть фичи `ads` по следующим причинам:

- во первых, ключ доступа к API и промпты никогда не должны попадать на клиент, поэтому он находится на сервере
- сервер должен организовать эндпоинты для 2х AI-функций

логика работы этих ручек завязана на концепцию объявлений. как минимум, без этого не получится написать нужный промпт и входные параметры для них. поэтому, используем `/ads/suggestions/*`

- аналогично, на клиенте, чтобы сформриовать запрос к ним, нужно знать об объявлениях.

Альтернативой было бы создание отдельного эндпоинта для всех ai-функций (`/ai/completions/*`, например), но тогда промпты пришлось бы формировать на клиентской стороне. Иногда они могут содержать чувствительную информацию, поэтому лучше так не делать
